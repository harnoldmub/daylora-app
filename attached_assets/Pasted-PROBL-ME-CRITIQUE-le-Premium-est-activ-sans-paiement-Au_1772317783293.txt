PROBLÈME CRITIQUE : le Premium est activé sans paiement.

Aujourd’hui, un utilisateur peut :
- choisir un template premium
- accéder aux fonctionnalités premium
- finaliser l’inscription
… sans avoir payé.

C’est interdit.

OBJECTIF :
Mettre en place un verrouillage strict :
✅ Premium = activé UNIQUEMENT après confirmation Stripe (paiement réussi).
❌ Aucun accès premium avant paiement confirmé.

RÈGLES À APPLIQUER :

1) Source de vérité
- Le statut premium doit venir UNIQUEMENT du backend (DB) mis à jour par webhook Stripe (payment_succeeded).
- Le front ne doit jamais “activer” premium localement.

2) Onboarding / inscription
- Si l’utilisateur choisit Premium pendant l’onboarding :
  -> Il doit être redirigé immédiatement vers le paiement.
  -> Tant que le paiement n’est pas confirmé :
     - On bloque la fin de l’inscription (pas de “Terminer”)
     - On affiche un état “En attente de paiement”
     - L’utilisateur ne peut pas entrer dans les zones premium.

3) Templates premium
- Autoriser la PREVIEW des templates premium (voir, tester visuellement)
- Interdire l’APPLIQUER/ACTIVER sans paiement confirmé.
- Si l’utilisateur clique “Utiliser ce template premium” :
  -> ouvrir le checkout (149€ paiement unique)
  -> puis seulement après confirmation, appliquer le template.

4) Gating des features premium
- Toutes les features premium doivent être protégées côté backend + côté front.
- Backend : toute route premium doit vérifier plan == premium.
- Front : masquer/disabled + CTA “Débloquer Premium”.
- Ajouter messages explicites : “Cette fonctionnalité est réservée à Daylora Premium (paiement unique 149€)”.

5) Paiement one-shot annuel (149€)
- Le plan annuel doit être paiement unique (pas subscription)
- Premium actif pour 12 mois après paiement réussi.
- En cas d’échec ou abandon checkout :
  -> premium reste OFF
  -> l’utilisateur revient sur une page “Paiement non finalisé”.

6) Validation & tests obligatoires
Cas à tester :
- User free -> tente template premium -> doit payer -> premium OFF tant que webhook pas reçu
- Paiement réussi -> premium ON + accès complet
- Paiement abandonné -> premium OFF
- Webhook retardé -> état "en attente" + bouton "Rafraîchir le statut"
- Tentative d’accès direct à une route premium -> 403 + message propre

LIVRABLES :
- Correction gating complète (front + backend)
- Liste des zones premium protégées
- Liste des fichiers modifiés
- Checklist de tests manuels

Ne pas refactor l’architecture.
Correction prioritaire P0.
